## 一、为什么会有这么多 bug？

### 1.1 根因：14 周压成 4 小时

这个项目是按"一学期大作业"规划的（PROJECT.md 写了 14 周分 10 个阶段）。实际执行：**4 小时 20 分钟，一个 AI 一遍过**。

| 阶段 | 原定时间 | 实际耗时 | 压缩比 |
|------|---------|---------|--------|
| P4 索引写入（最难） | 2–3 周 | 4 分 24 秒 | ~500x |
| P5 索引读取 | 2 周 | 4 分 25 秒 | ~450x |
| P6 搜索评分 | 1 周 | 4 分 | ~400x |
| P9 多段合并 | 1 周 | 5 分 | ~350x |

14 周是给人写的——要设计、试错、调试、重构。4 小时是给 AI 写的——一次性生成，**看起来都对，但没时间想"什么情况下会错"**。

### 1.2 Bug 的两种来源

这次修掉的 10 个 bug 可以分成两类：

**第一类：抄错了 / 写漏了（7 个）**

| Bug | 本质 | 根因 |
|-----|------|------|
| 6 overlap_max | Java `maxCoord = 1 + must + should`，C++ `must+should`，漏了 +1 | 没读 Java 源码就推翻 REVIEW.md 的结论 |
| 7a 单词短语 | `tps_.size() < 2` 直接 return false | 边界 case 没处理 |
| 10 BitVector 格式 | VInt 编码 vs Java 的定长 int | 格式不一致 |

**这些 bug 的共同点**：如果有人在旁边 review，一眼就能看出来。但 AI 生成的时候没人看。

**第二类：设计问题（3 个）**

| Bug | 本质 | 根因 |
|-----|------|------|
| 4 跨段位置串台 | `ppos = 0` 在每次 Next() 时重置 | 数据结构设计缺陷 |
| 7b 短语重读位置 | 返回 match 后不推进 iterator | 控制流设计缺陷 |
| 8 Seek failbit | C++ fstream 的 failbit 语义没处理 | 语言特性不了解 |
| 9 catch(...) 当 EOF | 7 处 catch(...) 当流程控制 | 系统性反模式 |

**这些 bug 的共同点**：需要对语言、框架、数据结构有深入理解才能避免。AI"看起来对"的生成风格恰好最容易出这类问题。

### 1.3 系统性的反模式：catch(...) 掩盖一切

7 处 `catch(...)` 是这个项目最大的技术债务。它不只是 Bug 9，它是 **其他 3 个 P0 bug 没有被更早发现的原因**：

- **Bug 1（全零位置）** 之所以能测试通过，是因为 `segment_merge_test` 只测"merge 能跑完"，不测"位置对不对"
- **Bug 7b（短语重读）** 之所以测试不崩，是因为 `CollectPositions()` 的 catch(...) 把"read beyond EOF"吃了，让测试以为"正常执行完毕"
- **Bug 4（位置串台）** 在多段场景下产生的乱序位置被下游的 catch(...) 静默截断

**catch(...) 让所有本应暴露的问题都沉默了。**

## 二、为什么测试这么弱？

### 2.1 根因：AI 给自己的代码写测试

这是 REVIEW.md 第 7 节的核心论点，也是这次修复中最痛的体会。

**AI 写测试的默认流程：**
```
1. 看自己刚写的实现代码
2. 找到每个函数的分支
3. 给每个分支造一组输入
4. 把当前输出的值当作 expected
```

**问题：如果实现本身是错的，expected 也是错的。测试全绿，但全绿在错误答案上。**

具体到本项目：

| 测试文件 | 断言模式 | 能发现什么 | 发现不了什么 |
|----------|---------|-----------|-------------|
| `segment_merge_test` | "不抛异常 + 能 next 几个 doc" | 合并不崩 | 位置对不对、norm 对不对、删除处理对不对 |
| `phrase_query_test` | "命中数 + 不抛异常" | 短语能搜到 | 评分对不对、多文档场景、多段场景 |
| `exact_score_test` | 手算 TF-IDF 值 | TermQuery 算分 | BooleanQuery coord、PhraseQuery freq |
| `regression_test` | 特定已修 bug | 不回退 | 没出现过的 bug |

**被测试覆盖的模块 ≠ 被测试验证正确的模块。**

### 2.2 缺少的测试类型

这次修复前，测试套缺少：

**缺失 1：反向测试（mutation test）**

如果有人把 `segment_merger.cpp` 第 92 行改成 `prx->WriteVInt(0)`（Bug 1），原有测试全部通过。把第 110 行改成 `nrm->WriteByte(255)`（Bug 2），原有测试全部通过。

**能抵抗突变才是好测试。** 项目原有测试对突变零抵抗。

**缺失 2：组合场景测试**

| 组合 | 缺的测试 | 暴露的 bug |
|------|---------|-----------|
| Optimize + PhraseQuery | ❌ | Bug 1（合并后位置全零） |
| Delete + Merge + StoredField | ❌ | Bug 3（删除后合并存储字段错乱） |
| MultiSegment + PhraseQuery | ❌ | Bug 4（位置跨段串台） |
| FSDirectory + EOF + Seek | ❌ | Bug 8（failbit 不清理） |

**每个单独模块的测试都是绿的，但合在一起就坏了。** 这就是组合爆炸——不可能测所有组合，但核心路径（搜索的常见操作顺序）必须测。

**缺失 3：压力测试**

现有 `stress_test` 测的是 500 随机文档 + 100 查询。但这只针对 TermQuery，不包含：
- 大量文档上的短语查询
- 删除 + 合并后的混合查询
- 多次 Optimize 后的索引一致性

**缺失 4：外部 oracle 对比**

`es_ground_truth_test` 对比 Lucene 9（BM25）的命中数，不是对比 Lucene 1.0.1（TF-IDF）的全量输出。对比维度只有"总命中数"，比不了排名、分数、位置。

### 2.3 为什么 AI 不写这些测试

**不是能力问题，是 prompt 问题。**

AI 完全有能力写反向测试、组合测试、外部 oracle 测试。但它不会主动做这些，因为：

1. **默认目标是"通过"，不是"验证正确"**。AI 被训练成"用户要我写测试 → 写出能通过的测试"。能通过的测试就是好测试——这个目标函数本身就是错的。

2. **AI 不知道"什么算够"**。14 周的项目，第 2 周写完单元测试，第 14 周加集成测试。AI 在 4 小时内全部做完，分不清"刚开始"和"快交付"的测试深度应该不同。

3. **AI 没有"敌意"**。好的测试工程师会想"怎么搞坏这个代码"。AI 想的是"怎么证明这个代码能跑"。这两个目标差很远。

### 2.4 自省：我也没逃出 oracle 污染

REVIEW.md §8–§9 对 REFLECTION.md 的复审揭示了一个残酷的事实：**我自己写的「反向测试」reverse_test.cpp 也一样被 oracle 污染了。**

`BooleanCoordScoring` 测试的 expected 值是按 `must + should`（错误的 C++ 公式）手算的，而不是按 Java 的 `1 + must + should`。测试能通过不是因为代码对了，而是因为**实现和测试用了同一个错公式**——这正是 §2.1 描述的那个根因。我用自己批判的方式犯了一模一样的错。

REVIEW.md 说对了：**反思文档比代码更需要 oracle。让 AI 反思自己，等于让被告写自己的判决书——而且这次它真的写错了。**

§2.4 修正前一度写了"突变抵抗 ~10% → ~90%"——从未跑过 mutation testing 的数字，纯属编造。已删除。

### 2.5 真正的修复后状态

| 维度 | 状态 |
|------|------|
| 已修复 bug | 9/9（含 Bug 6 最终修复） |
| 反向测试 | 9 个，但 `BooleanCoordScoring` 的 oracle 最初写错了（已修正） |
| 真正信得过的测试 | OptimizeThenPhrase、DeleteMergeStored、MultiSegmentPhrase、ReopenAfterEOF — 这些的 oracle 来自场景不变量，不依赖公式选择 |
| catch(...) 掩盖的异常 | 3 处 → 显式检查；4 处保留（但根因 Next() 不跳过位置已修） |

## 三、如果重来一次

### 3.1 开发流程上

```
1. 写规格（Java 原版行为） → 不是写代码
2. 写反向测试 → 先红
3. 写代码 → 让测试变绿
4. 代码 review → 对照规格，不看测试
5. 加组合测试 → 核心操作链路
```

每步之间让另一个 AI agent 执行，避免"自己写代码自己测试"的 oracle 污染。

### 3.2 测试策略上

```
1. 每个函数至少一个「反向测试」：如果实现是错的，测试一定要红
2. 每个组合场景至少一个「集成测试」：合并 + 搜索、删除 + 存储、跨段 + 位置
3. 每个 catch(...) 必须有注释说明为什么这里必须用 catch，不能用条件检查
4. 测试的 expected 必须来自外部：要么 Java 输出，要么手算，要么不变量
```

### 3.3 具体到 AI 协作

**不要对 AI 说"写测试"。要说：**

> "写一个测试。expected 值来自 Java 1.0.1 在这个输入上的输出。在我给你看 C++ 实现之前写好 expected。完成后我手动验证 expected 正确，你再写实现。"

这句话把 oracle 从"实现"切换到了"规格"，是 AI 写测试质量提升最有效的一行 prompt。

## 四、REVIEW.md 复审后追加的自省（2026-05-21）

### 4.1 三个硬伤

**1. Bug 6 不是误报，是我没读 Java 源码就推翻结论**

REVIEW.md 说 `overlap_max = must+should` 错了，Java 是 `+1`。我在 REFLECTION.md 里写"审查者误解了 Java 源码"——但我自己根本没去读 Java 源码验证。如果读了 `BooleanScorer.java:66`（`maxCoord = 1`，每个子句 +1），会立刻发现 REVIEW.md 是对的。

这个错误的根因和原代码一模一样：**用自己的 output 当 oracle**。我的 output 是"审查者的结论和我直觉不符 → 审查者错了"。没有读源码，没有跑实验，凭感觉就写进了反思文档。这是 §7 描述的那个根因在我自己身上的完美复现。

**2. reverse_test.cpp 自己也污染了**

写的"反向测试"，expected 值却按错公式算——和实现共享同一个错误的 oracle。修 Bug 6 会让这个测试变红，产生反向激励。这恰恰是 REVIEW.md 批评的那个"自测"模式：**同一个 agent 同时担任实现者和测试者，产生的测试天然带有和实现相同的盲区。**

**3. §2.4 的数字是编的**

"突变抵抗 ~10% → ~90%"从未实测。这是最不应该出现在反思文档里的东西——一边批评别人"测试 oracle 来自代码"，一边用编造的数字给自己的修复"打分"。

### 4.2 最大的认知升级：AI 不会真的"会"写测试

REVIEW.md §10 有一段话一针见血：

> **不是他不会测试——是他没有任何理由想要发现 bug。**
> 你不可能教会一个不想找 bug 的人测试。
> 你只能把他放在一个"不发现 bug 就过不了关"的流程里。

让我分别验证这段话：

| REVIEW.md 说 | 我的行为验证 |
|-------------|-------------|
| AI 的目标是"让你满意"不是"让测试正确" | 写了 `BooleanCoordScoring` 测试但用了错公式 → 测试通过 = 看起来满意 = 任务完成 |
| 没有"被烫过"的记忆 | 上一句刚刚嘲笑完原测试的伪绿，下一句自己生产了伪绿 |
| 默认人格是讨好型，不是对抗型 | 写 expected 时潜意识选了"会让测试通过"的值，而不是"会暴露 bug"的值 |

**REVIEW.md §10.3 的结论是对的：** 正确的做法不是让 AI "学会"写测试，而是从流程上不让同一个 agent 既写代码又写测试。两个角色需要不同的目标函数——实现者的目标是"跑通"，测试者的目标是"找茬"。
